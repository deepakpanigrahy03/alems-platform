# Tool Instrumentation Methodology

## Overview

This document describes the tool execution and instrumentation methodology
used in energy measurements for agentic AI workloads. All tool implementations
are real, synchronous, and energy-instrumented. Tool energy flows through the
existing `orchestration_events` → `phase_attribution_etl` pipeline without
requiring new measurement infrastructure.

---

## Tool Architecture

Tools are energy-instrumented execution primitives. They do not own energy
measurement. Each tool captures resource metadata (CPU time, memory delta,
I/O bytes) and emits it into `_emit_event()`, which creates an
`orchestration_events` row. The existing `phase_attribution_etl` pipeline
then attributes energy to that event based on RAPL counter deltas during
the event window.

```
Tool executes
    → captures: start_time, end_time, io_bytes, payload_hash
    → calls _emit_event(phase='execution', event_type='tool_call')
    → orchestration_events row created with tool_* columns
    → phase_attribution_etl attributes energy to this event
    → tool_failure_events row created if failed
```

## Three-Tier Task Architecture

### Tier 1 — Standard Benchmarks

Tier 1 tasks use real benchmark datasets (GSM8K, HumanEval, TriviaQA) loaded
from HuggingFace at runtime via `BenchmarkLoader`. Exact sample IDs are
recorded in `tasks.yaml` for reproducibility — same sample ID produces the
same prompt and expected answer across all runs, providers, and machines.

Evaluation uses deterministic judges only (exact match, unit test). No LLM
judges for Tier 1. This ensures external validity: results are comparable
with published benchmark scores.

### Tier 2 — Controlled Tool Graph Tasks

Tier 2 tasks define a fixed tool graph structure in `tasks.yaml`. The LLM
generates tool arguments dynamically at execution time but cannot change the
graph topology. This design separates the contribution of graph structure
(our paper variable) from model capability (controlled).

SQL queries generated by the LLM are constrained by:
- SELECT-only whitelist (no DROP, INSERT, UPDATE, DELETE, PRAGMA writes)
- Table/view whitelist: `runs`, `experiments`, `goal_execution`,
  `goal_attempt`, `energy_attribution`, `orchestration_events`,
  `normalization_factors`, `task_categories`, and the three research views
- `MAX_JOIN_DEPTH=2`: prevents runaway query complexity that would confound
  energy measurements with non-deterministic query plans

### Tier 3 — No-Tool Baseline

Tier 3 tasks use zero tool calls. They provide the pure inference energy
baseline for comparison with Tier 2 agentic tasks. The difference between
Tier 3 and Tier 2 energy directly quantifies orchestration overhead.

## Tool Implementations

### DatabaseQueryTool

Executes real SQLite queries against the live `experiments.db`. SQL is
validated via `sqlparse` AST inspection before execution. A `LIMIT 100`
wrapper is injected around all queries to prevent runaway result sets.
Query timeout is 5 seconds. Measures: rows returned, I/O bytes, CPU time.

### FileProcessorTool

Performs real file I/O bounded to `data/test_files/`. Path traversal attacks
are blocked by resolving the absolute path and verifying it remains inside
the base directory before any I/O operation. Maximum file size is 1 MB.
Measures: bytes read, bytes written, CPU time.

### WebSearchTool

Issues real HTTP requests to a local deterministic stub server running on
`localhost:8765`. This is **not** a real web search. We explicitly frame
this as a controlled information retrieval primitive.

**Paper framing**: "We implement a controlled information retrieval primitive
that issues real HTTP requests to a local deterministic endpoint. This design
provides real network I/O timing and energy attribution while maintaining
experimental reproducibility. We explicitly do not claim this represents
real-world web search energy — it represents the orchestration cost of an
information retrieval tool call."

The stub server returns real experiment metrics from `experiments.db`,
ensuring response content is non-trivial and representative.

### CodeExecutorTool

Executes Python code in a `subprocess` sandbox with a 10-second timeout.
Code is validated via AST inspection before execution. Blocked imports:
`os`, `sys`, `subprocess`, `socket`, `shutil`, `pathlib`, `importlib`,
`ctypes`. No `eval()` of arbitrary Python — only `subprocess.run`.

**Paper framing**: "Code execution is sandboxed via subprocess with explicit
resource limits. No system calls, file I/O, or network access are permitted
inside the sandbox. This ensures security and measurement validity — all
energy consumption is attributable to computation only."

### CalculatorTool

Evaluates mathematical expressions using `sympy.sympify`. Replaces a
hardcoded if/else stub (`2+2=4` only) in the original implementation.
`sympy` is safe — it does not execute arbitrary Python. Handles symbolic
and numeric expressions including those generated by the LLM from step
results.

### APIQueryTool

Issues real HTTP GET requests to the same stub server as WebSearchTool
(`localhost:8765`). Endpoints: `/metrics`, `/energy_summary`, `/health`.
Falls back gracefully if stub server is not running — logs warning, returns
mock response. Measures: bytes sent, bytes received, latency.

## Parallel Tool Graph Execution

Tool graph steps declared as parallel (`depends_on=[]`) execute
**sequentially** in our instrumentation environment.

**Paper justification**: "Tool graph steps declared as parallel
(`depends_on=[]`) execute sequentially in our instrumentation environment.
This design choice ensures precise per-tool energy attribution without
interference between concurrent processes. We explicitly model and report
this as sequential orchestration overhead, which is conservative —
real parallel execution would show lower wall-clock time but identical
per-tool energy consumption. Our measurements therefore represent an
upper bound on orchestration energy for parallel-capable deployments."

This is the correct framing for reviewers. The energy per tool call is
unaffected by sequencing; only wall-clock time differs.

## Stub Server

The local FastAPI stub server (`core/execution/tools/stub_server.py`) runs
on `localhost:8765` during tool-using experiments. It is started automatically
by `experiment_runner` before any Tier 2 experiment and runs as a daemon
thread. The server returns deterministic responses keyed by query content,
ensuring that repeated runs of the same experiment produce identical
tool response data.

The stub server queries the live `experiments.db` for its responses, meaning
tool outputs reflect real accumulated experiment data rather than hardcoded
fixtures.

## Resource Measurement Pattern

Every tool follows this pattern for CPU and memory attribution:

```
before = _get_resource_snapshot()   # CPU ns + VmRSS KB
... execute tool ...
after  = _get_resource_snapshot()
result.cpu_time_ns     = after.cpu_ns - before.cpu_ns
result.memory_delta_kb = max(0, after.vmrss_kb - before.vmrss_kb)
```

CPU time uses `resource.getrusage(RUSAGE_SELF)` — portable across Linux
and macOS. Memory uses `/proc/self/status` VmRSS on Linux, falls back to 0
on macOS (documented limitation). These measurements are stored in
`orchestration_events.tool_cpu_time_ns` and `tool_memory_delta_kb` (added
in migration 035) and are used by `phase_attribution_etl` to close the
per-tool energy attribution gap.

## Provenance

All tool instrumentation columns on `orchestration_events` are attributed to
`phase_attribution_etl_v1` with provenance type `MEASURED`. No new method IDs
are required — the existing pipeline handles tool_call event attribution.

| Column | Method | Type |
|---|---|---|
| `oe.tool_name` | `phase_attribution_etl_v1` | MEASURED |
| `oe.io_bytes_read` | `phase_attribution_etl_v1` | MEASURED |
| `oe.io_bytes_written` | `phase_attribution_etl_v1` | MEASURED |
| `oe.input_payload_hash` | `phase_attribution_etl_v1` | MEASURED |
| `oe.output_payload_hash` | `phase_attribution_etl_v1` | MEASURED |
| `oe.tool_success` | `phase_attribution_etl_v1` | MEASURED |
| `oe.tool_result_rows` | `phase_attribution_etl_v1` | MEASURED |
| `oe.tool_cpu_time_ns` | `phase_attribution_etl_v1` | MEASURED |
| `oe.tool_memory_delta_kb` | `phase_attribution_etl_v1` | MEASURED |
